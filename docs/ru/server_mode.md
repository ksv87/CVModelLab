# Серверный режим

[English version](../server_mode.md) | [README.ru](../../README.ru.md)

CV Model Lab работает в двух режимах:

- **Локальный автономный режим** (по умолчанию): все датасеты, предсказания и
  вычисления выполняются на вашем компьютере, как и раньше.
- **Опциональный серверный режим**: бэкенд на Python (FastAPI) просматривает
  датасеты в заданных разрешённых корнях, разбирает и оценивает данные COCO на
  стороне сервера, отдаёт байты изображений и миниатюры, считает AP-метрики и
  раздаёт сборку Flutter Web/PWA. Десктоп-приложение подключается к серверу;
  PWA, раздаваемая сервером, обращается только к своему источнику.

Сервер работает в режиме **только для чтения** по отношению к датасетам и
файлам предсказаний. Единственное, что он записывает, — это собственный кэш и
логи.

## Требования

- Python 3.9+
- [`uv`](https://docs.astral.sh/uv/) (рекомендуется)

## Установка и запуск сервера

```bash
cd server
uv venv
uv pip install -e ".[dev]"
cp server.example.yaml server.yaml   # затем отредактируйте allowed_roots
uv run python -m cvmlab_server.main --config server.yaml
```

Или используйте вспомогательные скрипты из корня репозитория:

```bash
scripts/run_server.sh server/server.yaml           # Linux/macOS
scripts/run_server.ps1 -Config server/server.yaml  # Windows
```

## Конфигурация

Сервер читает YAML-файл конфигурации. Основные поля:

```yaml
host: the current version.0
port: 8080

api_key: null            # задайте, чтобы требовать заголовок X-CVML-API-Key

allowed_roots:           # обязательно; единственные читаемые каталоги
  - id: datasets
    path: /data/datasets
    label: Datasets
  - id: experiments
    path: /data/experiments
    label: Experiments

project_manifests:       # опциональные серверные проекты-манифесты
  enabled: false
  directory: /data/cvmlab-projects

custom_server_paths:     # разрешить выбор файлов через обзор сервера
  enabled: true

cache:
  enabled: true
  directory: .cvmlab-server-cache

logs:
  directory: .cvmlab-server-logs

static_web:
  enabled: true
  root: ../build/web

cors:
  enabled: true
  allowed_origins:
    - http://localhost:*
```

`host`, `port` и `api_key` можно переопределить аргументами `--host`, `--port`
и переменными окружения `CVMLAB_HOST` / `CVMLAB_PORT` / `CVMLAB_API_KEY`.

### Разрешённые корни

Поле `allowed_roots` обязательно. Сервер читает файлы только внутри этих
каталогов. Обход путей (`..`), абсолютные пути за пределами корней и символьные
ссылки, ведущие наружу, отклоняются.

## Аутентификация и предупреждение об открытом доступе

- Если `api_key` задан, каждый запрос к `/api/` должен содержать заголовок
  `X-CVML-API-Key: <ключ>` — включая `/api/health` и `/api/config`. Без ключа
  отдаётся только статическое веб-приложение.
- Если `api_key` не задан, сервер работает в режиме **открытого доступа**: любой,
  кто может достучаться до хоста, сможет просматривать разрешённые корни и читать
  файлы датасетов.

Для защиты от случайного открытия:

- В интерактивном терминале сервер выводит предупреждение и просит подтверждение
  перед запуском без API-ключа. Ответ по умолчанию — **нет**.
- При неинтерактивном запуске (без TTY) сервер отказывается стартовать без
  API-ключа, если не передан флаг `--allow-unauthenticated`.

Значение API-ключа никогда не записывается в логи.

## Раздача PWA

Соберите веб-приложение Flutter и укажите путь в `static_web.root`:

```bash
flutter build web        # создаёт build/web
```

PWA будет доступна по адресу `http://<host>:<port>/`. Она автоматически
использует тот же источник для API, поэтому PWA, раздаваемая сервером,
обращается только к этому серверу. Если сборка отсутствует, API всё равно
работает, а на корневой странице показывается краткая подсказка.

Когда PWA открывает экран **Connect to Server**, она проверяет `/api/config` по
своему источнику и в зависимости от результата выбирает поведение поля адреса
сервера:

- **Раздаётся бэкендом CV Model Lab** (проверка успешна или возвращает `401`,
  потому что нужен ключ): адрес фиксируется на этом источнике, а поле доступно
  только для чтения — PWA общается лишь со своим сервером.
- **Standalone-режим** — сборка, открытая через `flutter run`, или размещённая
  отдельно от бэкенда (проверка не находит API): поле редактируемое, и адрес
  сервера вводится вручную, как в десктоп-приложении.

## Подключение из десктоп-приложения

1. На экране открытия выберите **Подключиться к серверу**.
2. Введите адрес сервера и, при необходимости, API-ключ.
3. При желании отметьте **Сохранить API-ключ для этого сервера**, чтобы запомнить
   его в локальных настройках (он привязан к адресу сервера и не записывается в
   файл проекта).
4. Нажмите **Проверить подключение**.

Затем откройте проект одним из способов:

- **Проект-манифест сервера** (если манифесты включены): выберите проект из
  списка, настроенного администратором.
- **Произвольные пути на сервере** (если включено): просмотрите разрешённые
  корни сервера и выберите JSON с аннотациями, каталог изображений и JSON с
  предсказаниями.

### Режим манифестов сервера

Администратор может разместить JSON-манифесты в каталоге
`project_manifests.directory`:

```json
{
  "schema_version": 1,
  "id": "traffic_lights_yolox",
  "name": "Traffic Lights YOLOX",
  "annotations_path": "/data/datasets/traffic/annotations/instances_val.json",
  "images_root_path": "/data/datasets/traffic/images",
  "model_runs": [
    {
      "id": "yolox_s_960_epoch_70",
      "name": "YOLOX-S 960 epoch 70",
      "predictions_path": "/data/experiments/run70/predictions.json",
      "ap_metrics_path": "/data/experiments/run70/ap_metrics.json"
    }
  ]
}
```

Все пути в манифесте проверяются на принадлежность разрешённым корням. Сервер
никогда не изменяет файлы манифестов.

## Файлы удалённых проектов

При открытии удалённого проекта можно сохранить локальный файл `.cvmlab.json` с
его описанием. В файле хранятся адрес сервера и либо идентификатор манифеста,
либо произвольные серверные пути, поэтому повторное открытие не требует заново
выбирать файлы. API-ключ в файле проекта не хранится.

Существующие локальные файлы проектов открываются без изменений.

## Кэш и логи

Единственные постоянные записи сервера:

- кэш разбора/индекса и результатов оценки,
- лениво создаваемые миниатюры,
- логи.

Кэш можно очистить запросом `POST /api/cache/clear`.

## AP-метрики

В серверном режиме AP-метрики COCO считаются на сервере. Если у запуска модели
указан `ap_metrics_path`, сервер загружает его; иначе вычисляет AP через
pycocotools. Десктоп-клиенту в удалённом режиме не нужны Python и pycocotools.

## Ограничения

- Серверный режим работает только для чтения: он не редактирует аннотации,
  предсказания и проекты и не загружает датасеты.
- Текущий удалённый клиент скачивает разобранный датасет, предсказания,
  сопоставления и метрики проекта, что хорошо подходит для малых и средних
  датасетов. Очень большие датасеты (сотни тысяч изображений) поддерживаются
  серверными постраничными и поизображенческими эндпоинтами, которые будущая
  версия клиента задействует напрямую.
- Многопользовательские аккаунты, редактирование и загрузка данных вне области
  задач.
